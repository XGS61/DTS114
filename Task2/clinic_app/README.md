# Clinic Appointment Flask App

Generated Flask API and website for the DTS114 Software Component.

## Purpose

This prototype supports appointment administration only. It does not provide diagnosis, treatment advice, prescriptions, or real patient-record storage.

Appointment API routes are session-protected. Use a demo account through `/`, `/app/login`, or `POST /api/auth/login`, or create a prototype account through `POST /api/auth/register`, before creating, listing, opening, or summarising appointments. Doctor accounts are linked to one doctor profile and can see only that queue; admin accounts can see every queue.

The login page displays `static/generated_clinic_image.png`, which is generated by the Task 1 notebook through APIFREE when configured and falls back to an embedded generated PNG when the image API is unavailable. The patient dashboard also displays fictional doctor roster photos generated through the same Task 1 APIFREE workflow when a valid key is configured.

The runtime data store is SQLite. The app automatically creates `data/clinic.sqlite3` on startup and seeds demo users, fictional doctors, and weekday shifts. Runtime database files are ignored by Git so the submitted repository remains a clean generated prototype rather than a real patient-record store.

Doctor accounts can update their own profile fields and upload a replacement profile photo. Uploaded runtime photos are stored under `static/uploads/doctors/` and ignored by Git; Task 1 still regenerates the three submitted demo doctor photos through APIFREE into `static/doctors/`.

## Local Setup

From the repository root, the recommended coursework environment is:

```bash
conda env create -f environment.yml
conda activate dts114-clinic-generator
```

The app can also be run with a standard Python virtual environment using `requirements.txt`:

```bash
python -m pip install -r requirements.txt
python app.py
```

Open `http://127.0.0.1:5000/` or `http://127.0.0.1:5000/app/login`.

## Demo Accounts

| Role | Username | Password |
|---|---|---|
| Patient | `patient` | `patient123` |
| Doctor | `doctor` | `doctor123` |
| Admin | `admin` | `admin123` |

Demo passwords and self-registered passwords are checked with Werkzeug password hashing. They are coursework demonstration credentials, not production authentication.

Self-registration is available from a separate Sign Up page linked below the Sign In form. The user chooses a patient, doctor, or admin prototype role during registration. Doctor registration creates a linked fictional doctor profile, while admin can create additional doctor accounts and patient-visible doctor profiles. This remains a coursework authentication prototype rather than production identity management.

## API Endpoints

| Method | Endpoint | Purpose |
|---|---|---|
| GET | `/health` | Service health check |
| GET | `/` | Public login page for deployment screenshots |
| GET | `/app/login` | Namespaced login page |
| POST | `/api/auth/login` | Start a demo role session |
| POST | `/api/auth/register` | Create a prototype patient, doctor, or admin account with a hashed password |
| POST | `/api/auth/logout` | End the current demo session |
| GET | `/api/auth/session` | Inspect the current demo session |
| GET | `/api/availability?date=YYYY-MM-DD` | Backward-compatible availability preview for the default doctor |
| GET | `/api/doctors?date=YYYY-MM-DD` | Show fictional doctors scheduled for a selected future weekday |
| GET | `/api/doctors/today` | Show today's fictional doctor roster and generated photo paths |
| GET | `/api/doctors/<doctor_id>/availability?date=YYYY-MM-DD` | Preview doctor-specific slot capacity and availability |
| GET | `/api/schedule/day?date=YYYY-MM-DD` | Staff day-board; doctor sees only own doctor slots, admin sees all |
| GET/PATCH | `/api/doctors/me` | Doctor-only profile, department, room, shift, capacity, and focus management |
| POST | `/api/doctors/me/photo` | Doctor-only upload for a replacement profile photo |
| GET/POST | `/api/admin/doctors` | Admin-only doctor account/profile list and creation |
| PATCH | `/api/admin/doctors/<doctor_id>` | Admin-only doctor profile update |
| POST | `/api/appointments` | Create pending appointment request |
| GET | `/api/appointments` | List visible appointment requests with `status`, `date`, `q`, `page`, and `page_size` filters |
| GET | `/api/appointments/<id>` | Get one appointment |
| PATCH | `/api/appointments/<id>/review` | Staff-only review action |
| PATCH | `/api/appointments/<id>/cancel` | Patient cancellation for the creator of the request |
| PATCH | `/api/appointments/<id>/reschedule` | Patient pending-request or staff-reviewed reschedule action |
| GET | `/api/appointments/<id>/audit` | Staff-only audit trail |
| GET | `/api/appointments/<id>/summary` | Non-diagnostic summary |
| GET | `/api/meta/requirements` | Generated requirements metadata |

## Scheduling Rules

- Appointment dates must be future weekdays.
- Each fictional doctor has seeded weekday shifts, room, department, slot length, and slot capacity.
- Clinic hours are bounded by the selected doctor's shift.
- Appointment times must use 30-minute increments.
- Appointment types are limited to configured administrative booking types.
- Conflict flags are recalculated by doctor, date, time, and capacity after create, review, cancellation, and reschedule actions.
- Create, review, cancel, reschedule, and conflict changes are written to the audit trail.

## Deployment

The app includes a `Dockerfile` for containerised deployment. Build and run it locally:

```bash
docker build -t dts114-clinic-app .
docker run --rm -p 5000:5000 -e FLASK_SECRET_KEY=local-docker-secret dts114-clinic-app
```

Open `http://127.0.0.1:5000/`.

For cloud evidence, create a Render Web Service using Docker runtime, root directory `Task2/clinic_app`, Dockerfile path `./Dockerfile`, Docker context `.`, and health check path `/health`.

Local Docker and Render Docker use the same Dockerfile. The local container is accessible on `http://127.0.0.1:5000/`; the Render container is accessible through the public Render URL after deployment.

The app also includes `render.yaml` and `runtime.txt` as a native Python Render fallback. The Docker deployment is the stronger containerisation evidence.
