# Clinic Appointment Flask App

Generated Task 2 Flask API and website for the DTS114 Software Component.

Current app version: `v1.3.1`

## Purpose

This application is a role-based clinic appointment administration prototype. It supports appointment booking, doctor/admin review, scheduling visibility, audit events, and non-diagnostic administrative summaries.

It is not a clinical system. It does not support diagnosis, treatment advice, prescriptions, emergency triage, or real patient records.

## User Workflows

| Role | Workflow |
|---|---|
| Patient | Sign in or register, choose a future weekday, choose a fictional doctor, select an available slot, submit an appointment request, track status, cancel own requests, and request pending reschedules |
| Doctor | Sign in to a linked doctor account, view only own queue, confirm/cancel/reschedule own appointments, view own day schedule, view audit events, update profile and upload an English-controlled profile photo |
| Admin | View all appointment queues, review all appointments, create doctor accounts, edit doctor profiles, and manage patient-visible doctor information |

## Runtime Storage

The app uses SQLite through Python's standard `sqlite3` library.

| File | Purpose | Git policy |
|---|---|---|
| `data/clinic.sqlite3` | Runtime database created automatically | Ignored |
| `data/.gitkeep` | Keeps the data folder in the repository | Committed |
| `static/uploads/doctors/` | Runtime doctor profile photo uploads | Ignored except `.gitkeep` |

On startup, the app creates database tables and seeds demo users, fictional doctors, and weekday shifts.

## Demo Accounts

| Role | Username | Password | Access boundary |
|---|---|---|---|
| Patient | `patient` | `patient123` | Own appointment requests only |
| Doctor | `doctor` | `doctor123` | Linked doctor queue only |
| Admin | `admin` | `admin123` | All queues and doctor management |

Self-registration is also available. Passwords are hashed with Werkzeug. The authentication design is for coursework demonstration and should not be treated as production identity management.

## API Reference

### Authentication

| Method | Endpoint | Purpose |
|---|---|---|
| `POST` | `/api/auth/login` | Start a role-based session |
| `POST` | `/api/auth/register` | Register a prototype patient, doctor, or admin account |
| `POST` | `/api/auth/logout` | End the current session |
| `GET` | `/api/auth/session` | Inspect current session state |

### Doctor And Schedule

| Method | Endpoint | Purpose |
|---|---|---|
| `GET` | `/api/doctors?date=YYYY-MM-DD` | List scheduled fictional doctors and availability summary |
| `GET` | `/api/doctors/today` | List today's doctor roster and generated photo paths |
| `GET` | `/api/doctors/<doctor_id>/availability?date=YYYY-MM-DD` | Return available, occupied, and conflicting slots |
| `GET` | `/api/schedule/day?date=YYYY-MM-DD` | Staff day-board, scoped by doctor/admin role |
| `GET` | `/api/doctors/me` | Doctor-only current profile |
| `PATCH` | `/api/doctors/me` | Doctor-only profile and shift update |
| `POST` | `/api/doctors/me/photo` | Doctor-only profile photo upload |
| `GET` | `/api/admin/doctors` | Admin-only doctor account/profile list |
| `POST` | `/api/admin/doctors` | Admin-only doctor account/profile creation |
| `PATCH` | `/api/admin/doctors/<doctor_id>` | Admin-only doctor profile update |

### Appointments

| Method | Endpoint | Purpose |
|---|---|---|
| `POST` | `/api/appointments` | Create a pending appointment request |
| `GET` | `/api/appointments` | List visible appointments with `status`, `date`, `q`, `page`, and `page_size` filters |
| `GET` | `/api/appointments/<id>` | Get a visible appointment |
| `PATCH` | `/api/appointments/<id>/review` | Doctor/admin review decision with note |
| `PATCH` | `/api/appointments/<id>/cancel` | Patient cancellation for own visible request |
| `PATCH` | `/api/appointments/<id>/reschedule` | Patient pending reschedule or staff reschedule with note |
| `GET` | `/api/appointments/<id>/audit` | Staff-only audit trail |
| `GET` | `/api/appointments/<id>/summary` | Non-diagnostic administrative summary |

### Metadata

| Method | Endpoint | Purpose |
|---|---|---|
| `GET` | `/health` | Service health, storage type, and version |
| `GET` | `/api/meta/requirements` | Generated requirements and project metadata |

## Scheduling Rules

- Appointment dates must be future weekdays.
- Appointment times must align to 30-minute increments.
- Appointment time must fall inside the selected doctor's shift.
- Appointment type must be one of the configured administrative appointment types.
- Capacity is checked per doctor, date, and slot.
- Conflicts are recalculated after create, review, cancellation, and reschedule actions.
- Audit events are written for create, review, cancel, reschedule, and conflict changes.

## Example API Session

```bash
curl -c cookies.txt -X POST http://127.0.0.1:5000/api/auth/login \
  -H "Content-Type: application/json" \
  -d "{\"username\":\"patient\",\"password\":\"patient123\"}"

curl -b cookies.txt "http://127.0.0.1:5000/api/doctors?date=2026-06-12"

curl -b cookies.txt -X POST http://127.0.0.1:5000/api/appointments \
  -H "Content-Type: application/json" \
  -d "{\"patient_name\":\"Alex Chen\",\"contact_email\":\"alex@example.com\",\"preferred_date\":\"2026-06-12\",\"preferred_time\":\"09:30\",\"doctor_id\":\"amelia-hart\",\"appointment_type\":\"General consultation booking\",\"reason\":\"Administrative booking request\"}"
```
